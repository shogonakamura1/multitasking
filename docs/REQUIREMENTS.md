# 並行作業ダッシュボード 要件定義書（ドラフト v0.1）

> 作成日: 2026-05-20 / ステータス: ドラフト（未決事項あり、§11参照）

---

## 1. 背景・課題

複数プロジェクトを並行で進める際、AI（Claude Code 等）への指示を出して**応答を待っている間に別プロジェクトへ移動する**。このコンテキストスイッチが繰り返されると:

- 人間側の認知リソースがあちこちに分散し、**疲労が蓄積する**
- 「このプロジェクトで**何をやっていたか / 次に何をするか**」を忘れる
- 並行している作業全体の**現在地（誰が何待ちか）が俯瞰できない**

「待ち時間に別の作業へ移る」こと自体は効率的だが、**復帰コスト（思い出すコスト）が高すぎる**のがボトルネック。

## 2. 目的・ゴール

サブディスプレイの約半分に**常時表示**し、視線を移すだけで以下が即座に分かる状態をつくる:

- **今やっていること** と **次にやること**
- **今日やること** のうち **未完了で着手が必要なもの**
- 各並行作業が **どの状態か**（特に「AI実行中＝待ち」か「自分の番＝要対応」か）

ゴール指標（定性）:
- 別プロジェクトから戻ったとき、**画面を1〜2秒見るだけで作業を再開**できる
- 「何やってたっけ？」と頭で思い出す作業をゼロにする

## 3. ターゲット・利用シーン

- **ユーザー**: 本人（シングルユーザー、ローカル利用前提）
- **シーン**: 2画面環境。メイン画面で実作業、サブ画面の半分にこのアプリを常時最前面表示
- **同時並行数**: 数件〜十数件のプロジェクト/タスクを想定

## 4. コア概念・データモデル

2階層（プロジェクト > タスク）で管理する。

### Project（プロジェクト）
| 項目 | 内容 |
|------|------|
| id | 一意ID |
| name | プロジェクト名 |
| color / icon | 視認性のための色・アイコン |
| status | `active` / `paused` / `done` |
| workdir | 紐づくディレクトリパス（hooks/プロセス監視用、任意） |
| order | 表示順 |

### Task（タスク）
| 項目 | 内容 |
|------|------|
| id | 一意ID |
| projectId | 所属プロジェクト |
| title | タスク名（今やっていること） |
| status | `todo` / `in_progress` / `waiting_ai` / `blocked` / `done` |
| nextAction | **復帰時にやること**（未来の自分へのメモ。最重要フィールド） |
| note | 補足メモ（LLMによる詳細付与の追記先） |
| dueToday | 今日やる対象か（boolean / 日付） |
| sources | 紐づく外部コンテンツの参照（Slack/Zulipスレッド・カレンダー予定・Google Tasks のID/URL。Phase 2） |
| updatedAt | 最終更新時刻 |

### 状態（status）の意味づけ ― 設計の肝
| status | 意味 | ダッシュボードでの扱い |
|--------|------|----------------------|
| `waiting_ai` | AIに指示を出して応答待ち | 「待ち」グループ。**完了検知の対象**。緑/点滅などで「終わったら自分の番」を示唆 |
| `in_progress` | 自分が手を動かしている | 「進行中」 |
| `blocked` | 他要因で止まっている | 要対応として強調 |
| `todo` | 未着手 | 今日やる一覧に表示 |
| `done` | 完了 | 折りたたみ/アーカイブ |

> `waiting_ai` ←→ `in_progress` の遷移（＝AIが終わって自分の番になる瞬間）が、本アプリの中心的なイベント。将来の自動検知（§9）もここを狙う。

## 5. スコープ（フェーズ分け）

入力方法は4種すべて将来的に取り込むが、**一度に全部は作らない**。確実に効く手動入力＋Claude Code連携をMVPに含め、自動検知は難易度が高いため最後に回す。

| フェーズ | 内容 |
|---------|------|
| **MVP（Phase 1）** | 常時表示UI＋手動入力＋プロジェクト/タスクの状態管理＋「今日やること」ビュー＋**Claude Code hooks連携**（状態の半自動更新・完了通知の起点＝自動検知L1も実質ここで成立） |
| **Phase 2（次）** | 外部ツール連携（Notion / Google カレンダー）。今日やることの取り込み |
| **Phase 3（最後）** | 作業完了の自動検知（§9）。プロセス/ターミナル監視（L2）＋ 画面キャプチャ解析（L3）。難易度が高く、効果と重さを見極めてから着手 |

### スコープ外（当面やらない）
- マルチユーザー / チーム共有 / クラウド同期
- モバイルアプリ
- 高度なガントチャート・工数管理（既存PMツールの領域）

## 6. 機能要件

### 6.1 MVP（Phase 1・必須）

**手動管理コア**
- **F-01** プロジェクトのCRUD（名前・色・状態・workdir）
- **F-02** タスクのCRUD（title・status・nextAction・note・dueToday）
- **F-03** タスク状態のワンクリック切り替え（特に `waiting_ai` ⇄ `in_progress`）
- **F-04** **俯瞰ビュー**: 全プロジェクトの「今やっていること / 次にやること / 状態」を1画面で縦に並べる
- **F-05** **待ちビュー**: `waiting_ai` のタスクを上部に集約（「今AIが動いている作業」が一目で分かる）
- **F-06** **今日やることビュー**: `dueToday` かつ未完了のタスク一覧
- **F-07** 各タスクの **最終更新からの経過時間**表示（「3分前 / 25分待ち」など。放置検知の手がかり）
- **F-08** ローカル永続化（再起動で復元）

**Claude Code hooks 連携（＝自動検知L1）**
- **F-10** Claude Code の hooks（Stop/Notification等）から、対象プロジェクト（workdir基準）のタスクを「AI完了 → `in_progress`（自分の番）」に更新する受け口（ファイル監視 or ローカルHTTP）
- **F-11** AI完了時のOS通知（どのプロジェクトが自分の番に戻ったか）

### 6.2 Phase 2（次・外部ツール連携 + ローカルLLM enrichment）

**取り込み**
- **F-20** Google カレンダー / Google Tasks / Slack / Zulip から情報を取り込み（予定・タスク・関連メッセージ）
- **F-21** カレンダー予定・外部タスクを「今日やること」候補として反映

**ローカルLLMによる関連付け・詳細付与（中核）**
- **F-22** 取り込んだ外部コンテンツ（Slack/Zulipメッセージ、カレンダー予定等）を**ローカルLLM**で解析し、**関連すると判断した既存タスクに紐づける**
- **F-23** 紐づけ結果に基づき、対象タスクへ**詳細情報を自動付与**（例: 関連スレッドの要約、決定事項、リンク、期限を `note` / `nextAction` / 添付メタに追記）
- **F-24** 突き合わせ・重複回避（同一案件の二重登録を防ぐ）
- **F-25** LLMによる紐づけ/付与は**提案 → ユーザー承認**を基本とし、誤紐づけを防ぐ（自動適用は任意設定）

### 6.3 Phase 3（最後・作業完了の自動検知）
- **F-30** workdir に紐づくプロセス/コマンドの開始・終了監視 → 状態の自動推定（L2）
- **F-31** 画面キャプチャ + AI解析による完了検知（L3。採否含め実証後に判断）

## 7. UI / 画面要件

> **本アプリはUIが主役。** 「現在地を一目で掴むダッシュボード」と「詳細を編集できるタスク管理」の2役を、狭幅パネル1つで両立させる。

**前提**: サブディスプレイ約半分の幅／常時最前面／情報密度高め・スクロール最小。

### 7.1 レイアウト ― 常時二ペイン（マスター・ディテール）

左に全体一覧（マスター）、右に選択中タスクの詳細（ディテール）を常時表示する。視線移動だけで「全体の現在地」と「今の作業の中身」を同時に把握できる。

```
┌──────────────┬───────────────────────┐
│ マスター(一覧) │ ディテール(選択中タスク)  │
│              │                       │
│ ⏳ 待ち       │ ● ProjB  API設計        │
│  ● ProjA 25分 │   状態: ▶ 進行中  [変更]  │
│  ● ProjC 8分  │   次:  バリデーション追加 │  ← nextAction
│ ▶ 進行中      │   メモ: …               │
│  ● ProjB  ◀  │   sources: #api-design ↗ │  ← 外部連携(P2)
│ ☑ 今日 3件    │   経過: 12分前更新       │
│  □ ProjD …    │   [編集] [完了] [次へ進む]│
└──────────────┴───────────────────────┘
```

- マスターは状態グループ（待ち / 進行中 / 今日やること）で区切る
- 行クリックでディテールが切り替わる（選択中行はハイライト）

### 7.2 ビジュアル方向 ― ダーク基調 × 状態色アクセント

長時間の常時表示でも目が疲れにくい暗めの背景に、**状態色を主役のアクセント**として使う（情報＝色が意味を持つ、装飾色は最小）。

- **状態色（セマンティック）**: 待ち=緑 / 進行中=青 / blocked=赤 / todo=ニュートラル / done=減光
- 背景はダークのサーフェス階層（奥=パネル背景／手前=カード）でレイヤー感を出す
- 状態色は左ボーダー・ドット・バッジに限定し、面では使いすぎない（眩しさ回避）

### 7.3 UI要件

- **U-01** 状態ごとの色分け（§7.2のセマンティックカラー）
- **U-02** `nextAction`（次にやること）をディテールに常時・最優先表示 ― 復帰の核
- **U-03** 状態変更はワンクリック（特に `waiting_ai` ⇄ `in_progress`）、本文はインライン編集
- **U-04** **レスポンシブ退避**: 幅が一定以下になったら二ペインを解除し、一覧優先の単一カラム＋詳細はオーバーレイ/展開に切替（狭幅での破綻防止）
- **U-05** 経過時間バッジ（「25分待ち」等。長待ち・放置を視覚化、しきい値超で強調）
- **U-06** マスターは「待ち」を最上部に集約（今AIが動いている作業が一目で分かる）
- **U-07** キーボード操作（上下で選択移動、状態トグルのショートカット）で視線を切らずに回せる
- 設計トーン: テンプレ然としない情報設計。階層・コントラスト・状態色を意味的に使う（装飾過多にしない）

## 8. 入力・データ取り込み（手段別）

| 手段 | フェーズ | 仕組み（案） |
|------|---------|-------------|
| 手動入力 | MVP | UI上で直接編集 |
| Claude Code hooks | MVP | hooks が ローカルファイル/HTTPエンドポイントへ「project, event(start/stop), task」を書き込む → アプリが監視・反映 |
| 外部ツール + ローカルLLM | Phase 2 | Google カレンダー / Google Tasks / Slack / Zulip から取り込み → ローカルLLMが関連タスクへ紐づけ・詳細付与（F-22〜F-25） |

## 9. 作業完了の自動検知（未決 ― 検討用たたき台）

**結論: 単一方式に賭けず、信号を多層化することを推奨。** 安い・正確な信号から段階導入する。

| 層 | 方式 | 長所 | 短所 | 推奨フェーズ |
|----|------|------|------|------------|
| **L1** | **Claude Code hooks の完了イベント**（Stop hook 等） | 軽量・正確・実装が容易。hooks連携をMVPに含むため追加コストほぼゼロ | Claude Code の作業に限定 | **MVPに含む（F-10/F-11）** |
| **L2** | プロセス/コマンド終了監視 | Claude Code以外の長時間処理（ビルド/学習等）にも効く | 「どのタスクの完了か」の紐付けが曖昧になりがち | Phase 3 |
| **L3** | 画面キャプチャ + AI解析 | 最も汎用（どんなツールでも検知しうる） | 重い・誤検知・**プライバシー配慮必須**・常時解析のコスト | Phase 3（最後） |

**推奨ライン**: L1（hooks）はMVPで作り込み、「AI応答完了→自分の番」を確実に取る。これだけで本来の課題（待ち→復帰）の大半は解ける。汎用的な画面検知（L3）は難易度・重さが高いため最後に、効果を見極めてから判断。

> ❓ 要相談ポイント: L3 をどこまで本気で作るか。「Claude Code 中心の運用なら L1 で十分」と割り切る選択肢も含めて、後続で判断したい。

## 10. 非機能要件

- **N-01 常時最前面 / 常駐**: always-on-top、ピン留め、自動起動オプション
- **N-02 軽量**: 待機時CPUほぼ0、メモリ低フットプリント（常時表示のため）
- **N-03 ローカル完結 / プライバシー**: データはローカル保存。クラウド送信なし。特にL3画面キャプチャ採用時は**ローカル処理 or 明示同意**を厳守
- **N-04 起動の速さ・即復元**: 再起動しても状態を失わない
- **N-05 邪魔をしない**: 通知は過剰にしない（AI完了など重要イベントのみ）

## 11. 技術構成（案）

| レイヤ | 採用 | 備考 |
|--------|--------|------|
| シェル | **Tauri** | 軽量・ネイティブ寄り。always-on-top、グローバルショートカット、トレイ常駐に対応。将来の画面キャプチャ/プロセス監視もRust側で実装しやすい |
| フロント | **TypeScript + React** | 情報密度の高いUI・エコシステム。状態管理は軽量に（Zustand等） |
| 永続化 | **SQLite**（Rust側 `rusqlite`） | タスク数増加・sources紐づけ（Phase 2）に強い。単一ユーザー・ローカル |
| hooks受け口 | **ローカルHTTP**（Tauri内に軽量ローカルサーバ） | Claude Code hooks から `curl` 一発で叩ける。loopback限定でバインド |
| ローカルLLM | Ollama 等（Phase 2） | 外部コンテンツの関連付け・要約・詳細付与をローカル実行。クラウド送信なし。モデル/ランタイムは要選定 |

> Web/フロントの詳細スタイル: ダーク基調×状態色アクセント（§7.2）。テンプレ然としない情報設計を重視。
> M1 の詳細設計・タスク分解は `DESIGN_M1.md` を参照。

## 12. ロードマップ（マイルストーン）

1. **M1（MVP）**: Tauri雛形 + データモデル + 俯瞰/待ち/今日やること の3ビュー + 手動入力 + ローカル永続化 + always-on-top ＋ **Claude Code hooks連携（L1自動検知・完了通知）**
2. **M2（次）**: 外部ツール連携（Notion / Google カレンダー）
3. **M3（最後）**: 作業完了の自動検知 ― プロセス監視（L2）＋ 画面キャプチャ実証（L3、採否含め判断）
